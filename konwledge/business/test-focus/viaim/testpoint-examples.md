# 高质量测试点样例库

从 V2.0.2 搜索绑定流程的实际测试点中提取的设计模式，展示测试点的排列逻辑和设计思路，供 AI 生成测试点时参考。

> 来源：V2.0.2 设备绑定与未激活会员（经测试工程师审校后的版本）
> 最后更新：2026-04

---

## 样例1：流程按阶段编排——弹窗紧跟触发条件

**设计特征**：测试点按用户实际操作的时间线编排，每个弹窗/状态出现后，紧接着就是该弹窗的内容验证、操作验证、关闭方式验证，形成"触发→验证→处理→下一阶段"的自然闭环。

```markdown
## 搜索绑定流程

### tc-P0: 搜索中-网络正常-已绑定本账户设备-直接连接成功
### tc-P0: 搜索中-网络正常-设备未绑定-双耳在线-弹出绑定弹窗
### tc-P1: 绑定弹窗-展示设备图片和绑定文案-点击绑定-显示绑定中loading
### tc-P2: 绑定弹窗-X/顶部下拉关闭弹窗（不支持弹窗外关闭）-App未连接状态
### tc-P2: 绑定中状态-不可关闭弹窗-不可重复点击绑定按钮
### tc-P0: 绑定成功-首次连接该耳机-App显示已连接-弹出欢迎弹窗-走连接成功后弹窗流程
### tc-P1: 绑定成功-非首次连接该耳机-Toast设备绑定成功
### tc-P2: 绑定中网络异常-覆盖弹出网络不佳弹窗-App保持连接中动画
### tc-P3: 绑定中网络不佳弹窗-点击重试-进入绑定中
### tc-P3: 绑定中网络不佳弹窗-点击X/下拉关闭弹窗-回到绑定弹窗
```

**编排逻辑**：

```
搜索结果(115-116) → 绑定弹窗内容+关闭(117-118) → 绑定中防护(119)
  → 绑定成功(120-121) → 绑定中异常+弹窗处理(122-124)
```

**为什么好**：
- 不按优先级排列（P0/P1/P2/P3 混排），而是按**操作时间线**排列，符合执行测试的自然顺序
- 弹窗触发后紧跟该弹窗的所有验证（内容、按钮、关闭方式），不会在不同弹窗之间跳来跳去
- 异步操作的中间状态（绑定中loading）单独验证防护措施（不可关闭、不可重复点击）

**通用原则**：测试点排列应遵循**用户操作时间线**，而非按优先级分组。弹窗/状态出现后，立即跟上该弹窗的全部验证，再进入下一阶段。

## 样例2：异常分支紧跟正常流程——触发后就地展开

**设计特征**：每种异常场景的触发条件出现后，紧接着就是该异常弹窗的完整验证（按钮、关闭方式），形成"异常触发→弹窗验证→回到主流程下一个异常"的结构。

```markdown
### tc-P2: 设备已被其他账号绑定-弹出已被绑定弹窗-停止搜索
### tc-P3: 已被绑定弹窗-知道了/X/下拉关闭弹窗-弹窗外不可关闭
### tc-P3: 已被绑定弹窗-点击切换账号-跳转账号设置页
### tc-P2: 设备未绑定-其中一个耳机未在线-弹出需要连接两只耳机弹窗-停止搜索
### tc-P3: 需要连接两只耳机弹窗-点击重试-重新搜索
### tc-P3: 需要连接两只耳机弹窗-X/下拉关闭弹窗-弹窗外不可关闭
```

**编排逻辑**：

```
异常A触发(125) → A弹窗关闭(126) → A弹窗按钮(127)
  → 异常B触发(128) → B弹窗按钮(129) → B弹窗关闭(130)
```

**为什么好**：
- 异常触发条件与弹窗验证**紧密配对**，不会出现"所有触发条件在前，所有弹窗验证在后"的分离
- 每个异常分支是**自包含**的一组，方便在测试执行时按组跳过或重点执行
- 相似结构的弹窗（已被绑定/需要两只耳机/网络不佳）各自完整，对比后能发现关闭方式的差异

**通用原则**：异常场景的弹窗/提示验证，应紧跟在触发条件之后就地展开，不要集中到单独的"弹窗关闭验证"区域。

## 样例3：条件组合矩阵——网络×缓存×绑定状态

**设计特征**：系统性穷举多条件组合，标题结构完全统一，用条件差异来区分预期行为。

```markdown
### tc-P0: 搜索中-网络正常-已绑定本账户设备-直接连接成功
### tc-P0: 搜索中-网络正常-设备未绑定-双耳在线-弹出绑定弹窗
### tc-P1: 搜索中-网络异常-有缓存已绑定此账号-直接连接成功
### tc-P2: 搜索中-网络异常-有缓存未绑定此账号-弹出网络不佳弹窗-停止搜索
### tc-P2: 搜索中-网络异常-无缓存-弹出网络不佳弹窗
```

**组合矩阵**：

| 网络状态 | 缓存/绑定状态 | 预期行为 | 优先级 |
|---------|-------------|---------|--------|
| 正常 | 已绑定本账户 | 直接连接成功 | P0 |
| 正常 | 未绑定+双耳在线 | 弹出绑定弹窗 | P0 |
| 异常 | 有缓存已绑定 | 直接连接成功（离线降级） | P1 |
| 异常 | 有缓存未绑定 | 网络不佳弹窗+停止 | P2 |
| 异常 | 无缓存 | 网络不佳弹窗 | P2 |

**为什么好**：
- 标题结构统一：`搜索中-[网络状态]-[缓存/绑定条件]-[预期行为]`，一眼可对比差异
- 优先级递进合理：正常路径 P0 → 可降级 P1 → 异常阻断 P2
- 虽然分散在流程的不同位置（115-116在开头，131-133在中段），但标题结构的统一让它们作为矩阵仍然清晰可辨

**通用原则**：涉及多条件组合时，标题结构保持统一格式 `[场景]-[条件A值]-[条件B值]-[预期行为]`，即使分散在流程不同位置也能快速辨识同一矩阵的成员。

## 样例4：多入口覆盖——入口在后、按入口状态分组

**设计特征**：多入口测试点放在流程末尾（而非开头），先验证核心流程正确性，再逐入口确认入口可达性和状态差异。

```markdown
### tc-P0: 首页点击添加设备-进入搜索设备页面-可正常绑定并连接成功
### tc-P2: 解锁Basic会员弹窗-点击添加设备-进入搜索设备页面-可正常绑定并连接成功
### tc-P2: 我的viaim设备-点击添加设备-进入搜索设备页面-可正常绑定并连接成功
### tc-P2: 耳机设置页面-点击右上角添加设备-进入搜索设备页面-可正常绑定并连接成功
### tc-P2: 耳机设置页面-点击重新连接-可正常绑定并连接成功
### tc-P1: 自动回连-可正常绑定并连接成功
### tc-P1: 首页右上角-首次安装显示添加设备按钮-退出绑定后依然显示-连接成功后变为蓝牙状态图标
### tc-P1: 耳机设置页-未连接-点击添加设备-进入搜索设备页面
### tc-P1: 耳机设置页-已连接-点击添加设备-弹请先断开弹窗-支持X/下拉/弹窗外关闭弹窗
### tc-P2: 耳机设置页-搜索中状态-添加设备按钮置灰-点击无响应
```

**编排逻辑**：

```
主入口 P0(136) → 其他入口可达性(137-141)
  → 入口状态差异验证：首次安装(142) / 未连接(143) / 已连接(144) / 搜索中(145)
```

**为什么好**：
- 主入口（首页）标为 P0，其他入口标为 P2（降级为"可达性验证"），优先级区分合理
- 入口可达性验证在前，入口状态差异验证在后，形成"先确认入口通 → 再验证状态分支"的递进
- 耳机设置页的三种状态（未连接/已连接/搜索中）紧挨在一起，形成同一页面的状态矩阵

**通用原则**：多入口测试放在核心流程之后。主入口为 P0，其余入口降为 P2 验证可达性。同一入口的不同状态紧挨排列。

## 测试点编排总原则

从以上 4 个样例中提炼的编排规则：

| 原则 | 说明 |
|------|------|
| **时间线优先** | 按用户操作的自然时间线排列，不按优先级分组 |
| **弹窗就地展开** | 弹窗触发后紧跟内容/按钮/关闭验证，不集中到最后 |
| **异常紧跟正常** | 异常分支出现在对应正常流程之后，不单独分区 |
| **矩阵格式统一** | 同一条件组合的测试点标题结构统一，便于对比 |
| **多入口放末尾** | 核心流程验证完后，再逐入口确认可达性和状态差异 |
| **同页面状态紧邻** | 同一页面/组件的不同状态测试点紧挨排列 |
