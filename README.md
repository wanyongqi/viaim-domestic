# viaim 测试工作台使用指南

本项目基于 Cursor AI 辅助测试工作，覆盖测试用例编写、用例复查优化、回归重点建议、测试报告功能点梳理、现网包验证、完整测试报告生成六类工作场景。

---

## 二、七类工作场景与调用方式

### 场景 1：编写测试用例

**触发方式：** `@viaim-testing-spec.mdc` + 需求测试点

**标准流程：**
1. 整理需求测试点（参考各版本的需求点格式）
2. 在对话框输入 `@viaim-testing-spec.mdc`，粘贴需求测试点
3. AI 会先做需求分析并提问不明确的地方
4. 回答 AI 的问题后，等待 AI 完善分析
5. 确认分析无误后，说"开始生成"或"好"
6. 用例文件自动输出到 `xxx版本/test case/需求名称_测试用例.md`

**关键提示：**
- 测试点中需标注重点关注项（如"重点关注暂停后是否可以继续录音"），AI 会优先覆盖
- 不明确的需求点务必在 AI 提问阶段说清楚，不要等到用例生成后再改
- 涉及语种覆盖时，AI 会参考 `语种优先级_可读版.md`，无需额外说明
- 三类录音（通话/现场/音视频）默认分开写；如允许合并请明确告知

---

### 场景 2：复查与优化测试用例

**触发方式：** 直接说"复查 `@xxx_测试用例.md`" 或"优化这份用例"

**标准流程：**
1. AI 先输出全量审视结论（不直接改文件）
2. 问题按"高 / 中 / 低"分级整理
3. 确认或调整建议后，AI 再落到正式文档

**关键提示：**
- AI 默认"先审视、再改文件"，如需直接修改请明确说"直接修改"
- 复查只修复问题，不会新增未经确认的功能逻辑
- 三类录音用例不会被强行合并

---

### 场景 3：生成回归重点建议

**触发方式：** `@regression-focus.mdc` + "帮我分析本次迭代的回归重点"

**标准流程：**
1. 提供本次迭代的需求点或简约版测试点
2. AI 分析直接影响、间接影响和核心主流程必验项
3. 输出按"必回归 / 建议回归 / 可选回归"分级的建议清单

**输出格式示例：**
```md
## 回归重点建议（xxx 版本）

### 必回归
- **录音启停与暂停**（主流程，每版必回归）：核心功能，用户强感知

### 建议回归
- **语言选择逻辑**（间接影响）：本次改动触及公共语言选择器

### 可选回归
- **xxx 功能**：改动关联度弱，按资源决定
```

**关键提示：**
- 每版固定必验：三类录音启停、实时转写、老数据兼容、单条导出、蓝牙连接
- 如已有上一版本的回归建议文档，可以一并提供，AI 会在此基础上更新

---

### 场景 4：生成测试报告功能点

**触发方式：** `@test-report-summary.mdc` + "生成本次迭代的简约版测试点"

**标准流程：**
1. 提供已完成的测试用例文件，或本次迭代的需求点列表
2. AI 按功能模块提炼覆盖要点
3. 输出文件：`xxx版本/xxx简约版测试点.md`

**输出格式示例（参考 `630简约版测试点.md`）：**
```md
## 1. 录制过程中可暂停录制

- **基础控制**：覆盖三类录音的暂停、继续操作，以及转写状态的正确恢复。
- **异常与边界**：覆盖最小化返回、杀进程结束并保存、锁屏恢复等场景。
- **兼容性与UI**：浅色/深色/英文模式适配。
```

**关键提示：**
- 面向测试报告读者（PM / QA Leader），不要展开用例细节
- 每个功能模块控制在 3-8 条覆盖要点

---

### 场景 6：生成完整测试报告

**触发方式：** `@test-report-generation.mdc` + "生成本版测试报告"

**所需输入（三项缺一不可）：**
1. 缺陷 Excel 文件（需含 `严重程度` 字段，格式同 `iFLYBUDS--xxxx-xx-xx.xlsx`）
2. 功能点简约描述文件（由场景 4 生成的 `xxx简约版测试点.md`）
3. 版本号 + 测试周期

**标准流程：**
1. AI 运行 `parse_bugs.py` 解析 Excel，自动统计各端数量、严重等级分布、优先级分布、暂不修复列表
2. 读取功能点简约描述，插入报告"功能点评估"章节
3. 按 V6.3.0 报告结构生成完整 Markdown 报告
4. 输出文件：`xxx版本/测试报告_国内viaim_Vx.x.x.md`

**Excel 解析规则（已确认锁定）：**
- 平台识别：只取标题**第一个【】标签**，`【现网xxx】` 前缀保留计入总数
- `已关闭` / `已关闭（未修复）` → 已闭环（均计入总数）
- `严重程度` 字段格式：`1-致命` / `2-严重` / `3-一般` / `4-轻微`
- 仅统计 `工作项类型` 为 `缺陷` / `缺陷（产品/设计）` 的条目

**关键提示：**
- 确保 Excel 导出时包含 `严重程度` 字段（系统默认不导出，需手动添加列）
- 功能点文档由 `@test-report-summary.mdc` 先行生成，再调用本场景
- 参与人员如无变动，AI 默认沿用上版本

---

### 场景 7：生成 / 更新现网包验证清单

**触发方式：** `@production-verification.mdc` + "生成现网包验证清单" 或 "更新验证清单"

**标准流程：**
1. 提供本次迭代的需求点或简约版测试点
2. 如有上一版本的验证清单（如 `630现网包验证重点.md`），一并提供
3. AI 对照迭代内容更新清单（新增/调整/删除条目）
4. 输出文件：`xxx版本/xxx现网包验证重点.md`

**关键提示：**
- 每版固定必验项不可省略（AI 会自动兜底）
- 新功能和老数据兼容均需在清单中体现
- 可一次性生成，也可在已有清单基础上增量更新

## 三、完整版本迭代工作流

```
Step 1  整理需求测试点
         ↓
Step 2  @viaim-testing-spec.mdc → 生成各需求测试用例文件
         ↓
Step 3  复查用例 → 确认无漏测 → 落文档
         ↓
Step 4  @regression-focus.mdc → 生成回归重点建议 → 指导执行回归测试
         ↓
Step 5  @test-report-summary.mdc → 生成简约版测试点（功能点文档）
         ↓
Step 6  @test-report-generation.mdc → 提供缺陷Excel + 功能点文档 → 生成完整测试报告
         ↓
Step 7  @production-verification.mdc → 生成 / 更新现网包验证清单 → 上线前执行
```

## 四、缺陷 Excel 导出规范

每次迭代结束后，从缺陷管理系统导出 Excel 时，需确保包含以下字段：

| 字段名 | 说明 | 备注 |
|---|---|---|
| 标题 | 缺陷标题，含平台标签 | 必须，格式如 `【Android-x.x.x】xxx` |
| 状态 | 已关闭 / 暂不修复 / 待修复 | 必须 |
| 工作项类型 | 缺陷 / 缺陷（产品/设计） | 必须，用于过滤 |
| 优先级 | 紧急 / 高 / 中 / 低 | 必须 |
| 严重程度 | 1-致命 / 2-严重 / 3-一般 / 4-轻微 | **必须，系统默认不导出，需手动添加** |
| 迭代 | 国内移动端 Vx.x.x | 可选，用于过滤当前版本 |

> ⚠️ `严重程度` 字段系统默认不在导出列中，每次导出前需手动勾选该字段。

## 六、rules 与 skills 的区别

| 类型 | 位置 | 触发方式 | 作用 |
|------|------|----------|------|
| **rules** | `.cursor/rules/*.mdc` | 手动 `@` 引用 | 约束 AI 的行为原则和输出规范 |
| **skills** | `.cursor/skills/*/SKILL.md` | 自然语言自动识别 | 提供具体的操作流程和输出模板 |
| **00-language.mdc** | `.cursor/rules/` | 自动生效（alwaysApply） | 全局强制使用简体中文 |

> **说明**：rules 和 skills 配合使用，rules 定义"做什么、怎么做的原则"，skills 定义"具体操作步骤和输出格式"。通常 `@` 引用 rule 文件即可，AI 会自动加载对应 skill。

## 八、注意事项与常见坑

1. **需求分析阶段不要跳过提问**  
   AI 提问是为了避免用例生成后大面积返工，需求中的每一个不明确项都应在提问阶段确认清楚。

2. **说"好"或"开始生成"才会真正输出用例**  
   AI 分析完需求后会等待指令，直接说"好"即可触发生成。

3. **用例文件路径默认含 `test case` 子目录**  
   单需求文件输出到 `xxx版本/test case/需求名称_测试用例.md`，汇总文件输出到 `xxx版本/xxx测试用例.md`。

4. **优先级比例允许浮动，但不要为凑比例加删用例**  
   体量小的需求点不强求 P0/P1/P2/P3 全部出现。

5. **格式基准以 `用例输出要求.md` 为权威**  
   如遇 AI 输出格式与预期不符，可说"严格按照 `@用例输出要求.md` 的格式重新输出"。

6. **复查不等于直接修改**  
   AI 默认先出审视结论、等确认后再改文件；如需直接修改，明确说"直接修改"。

7. **涉及语种的需求点需参考语种优先级文件**  
   AI 会自动参考 `语种优先级_可读版.md`，无需每次手动说明，但可以提醒 AI 重点关注某些语种。
