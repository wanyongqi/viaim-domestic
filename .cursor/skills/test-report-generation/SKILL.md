# Test Report Generation

## Quick Start

当用户要求生成测试报告时，按以下顺序执行：

1. 向用户收集所需输入（见 Step 1），特别是**每次必问的四项**：参与人员、测试周期、版本号、风险评估原因。
2. 可选先跑 `python scripts/parse_bugs.py <xlsx路径>` 预览缺陷统计，给用户确认数字。
3. 调用 `scripts/generate_report.py` 一次性产出 .docx：
   ```powershell
   python scripts/generate_report.py \
     --version <版本号> --cycle "<测试周期>" \
     --excel <缺陷xlsx路径> --points <简约版测试点md路径> \
     --risk "<风险评估原因>" \
     --pm <产品> --design <视觉> --dev <研发,逗号分隔> --qa <测试,逗号分隔> \
     --out "xxx版本/测试报告_国内viaim_Vx.x.x.docx"
   ```
4. 输出文件：`xxx版本/测试报告_国内viaim_Vx.x.x.docx`

## Core Workflow

### Step 1：确认输入

向用户确认（未提供则逐项询问）：

**数据类（脚本自动处理）：**
- [ ] 缺陷 Excel 文件路径（格式同 `iFLYBUDS--xxxx-xx-xx.xlsx`，需含 `严重程度` 列）
- [ ] 功能点简约描述文件路径（`xxx简约版测试点.md`）

**每次必问类（脚本参数必填）：**
- [ ] 版本号（如 `6.4.0`，**不含 V 前缀**）
- [ ] 测试周期（如 `2026.4.14 - 2026.5.10`）
- [ ] 参与人员：产品 / 视觉 / 研发（多人） / 测试（多人）
- [ ] 风险评估原因（嵌入"遗留 N 个暂不修复问题，非【严重】缺陷，<原因>"）

**可选类（有默认值）：**
- [ ] 版本记录表记录人（默认 `万永琪`）
- [ ] 变更原因（默认 `V{版本号}版本`）
- [ ] 封面年月（默认当前月）
- [ ] 模板 .docx（默认 `630版本/测试报告_国内viaim_V6.3.0.docx`）

---

### Step 2：生成报告 .docx

调用 [`scripts/generate_report.py`](mdc:scripts/generate_report.py) 一次性产出完整 .docx：

```powershell
python scripts/generate_report.py `
  --version 6.4.0 `
  --cycle "2026.4.14 - 2026.5.10" `
  --excel "640版本/iFLYBUDS--2026-05-10.xlsx" `
  --points "640版本/640简约版测试点.md" `
  --risk "主要为xxx（从用户输入原文填入）" `
  --pm "李倩" `
  --design "唐淑娴" `
  --dev "徐康,郑文祥,孙学林,万宗咏,孙庆,吴胜,廖金龙" `
  --qa "万永琪,罗娜,马鑫鑫" `
  --out "640版本/测试报告_国内viaim_V6.4.0.docx"
```

脚本内部会：

1. 调用 `parse_bugs.load_bugs()` 读 xlsx，得到缺陷统计
2. 解析 `xxx简约版测试点.md`，按 `## 序号. 名称` + 无序列表的约定得到功能点列表
3. 以 V6.3.0 报告为模板 `python-docx` 打开，**原位替换**：
   - 封面：版本号、年月
   - 引言：参与人员四行
   - 测试版本信息：版本号、测试周期
   - 缺陷概览：总数、各端数
   - 风险评估：遗留数 + 用户原因
   - 总缺陷段：总数
   - 版本记录表：追加一行（序号自增、日期=今天、记录人=万永琪）
   - 测试范围表：清空数据行，逐条填入功能点
   - 缺陷等级分布表：严重程度×状态全部填数
   - 各端占比表：各端×优先级+占比填数
   - 已知问题表：清空数据行，逐条填入暂不修复缺陷（严重程度列填 致命/严重/一般/轻微）
4. 另存为 `--out` 指定路径

依赖：`openpyxl` + `python-docx`（见根目录 `requirements.txt`，首次用执行 `pip install -r requirements.txt`）。

**解析规则（锁定）：**

| 维度 | 规则 |
|---|---|
| 工作项筛选 | 仅保留 `工作项类型 ∈ {缺陷, 缺陷（产品/设计）}` |
| 平台识别 | 取标题**第一个【】标签**，匹配 Android / iOS / 服务端 / 未知 |
| 严重程度 | 按 `1-致命 / 2-严重 / 3-一般 / 4-轻微` 前缀或关键字归一化 |
| 状态归一 | `已关闭` / `已关闭（未修复）` → 已闭环；`暂不修复` → 暂不修复；`待修复` / `处理中` / `重新打开` → 待修复 |
| 功能点解析 | `xxx简约版测试点.md` 中 `## 数字. 名称` 为功能点名称，紧随的 `-`/`*` 列表项为覆盖要点 |

---

### Step 3：交付与校对

脚本运行成功后：

1. 打印摘要到 stdout（总缺陷数 / 各端数 / 暂不修复数 / 功能点数）—— 可直接贴给用户二次确认。
2. 打开生成的 .docx 复核 3 处**脚本无法自动推断、需人工/用户确认**的位置：
   - 已知问题表"**原因分析**"列：脚本留空，需用户按每个遗留问题补充
   - "整体质量评估"段落：脚本沿用上版本固定文案，如本版有差异需提醒用户调整
   - 版本记录表新追加行"**变更措施**"列（第 5 列）：脚本留空，用户若需要可自行填写

## 注意事项

- **统计数字来源唯一**：所有数字由 `generate_report.py` 从 xlsx 现场解析，不得手动估算或凑整。
- **功能点覆盖要点**：简约版测试点中 `-` / `*` 列表项整段原样插入"评估结果"列，不改写措辞。
- **遗留问题标题完整保留**：含版本号、模块名的完整标题（如 `【iOS-6.3.0】【翻译】...`）不得缩写。
- **版本记录追加不覆盖**：历史记录全部保留，脚本自动在表尾追加 1 行，序号自增、日期=今天、记录人默认"万永琪"。
- **参与人员每次必问**：参与人员四个参数（--pm / --design / --dev / --qa）均为必填，用户未提供时必须逐项询问。
- **风险评估原因每次必问**：--risk 为必填参数，用户未提供时必须询问，不得自行编造。
- **模板文件不得直接编辑**：`630版本/测试报告_国内viaim_V6.3.0.docx` 作为样式模板长期保留，修改需新建副本。
